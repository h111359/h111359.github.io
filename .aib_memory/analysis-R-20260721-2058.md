## Overview

- Request ID: R-20260721-2058
- Request title: Holistic strategic review of the personal website

### Background

The developer wants an evidence-grounded strategic critique of the current personal website so a later implementation effort can unify its professional, utility-project, and creative purposes without abandoning its static GitHub Pages model.

### Scope

The work creates `ai-critique.md`, targets approximately 3,000–5,000 words, treats the current workspace as authoritative for source scope and behavior, and supplements source inspection with representative live observations limited to routes linked from primary navigation. It assesses strategy, information architecture, visual design, UX, WCAG 2.2 AA accessibility without certification claims, public messaging, privacy, technical quality, performance, search/sharing readiness, portfolio presentation, and creative presentation. The output must include a coherent comprehensive-redesign direction, impact-versus-effort priorities, phasing, risks, and a conclusion.

### Out of scope

Implementation, edits to existing website artifacts, replacement code, file-by-file planning, framework/build/backend/database adoption, penetration testing, formal WCAG certification, exhaustive copy replacement, and review of public routes not linked from primary navigation are excluded.

### Industry Knowledge

- WCAG 2.2 organizes testable accessibility criteria under perceivable, operable, understandable, and robust principles; AA conformance requires satisfying all A and AA criteria across each full responsive page. Applicability: the critique should distinguish obvious static risks from items requiring keyboard, screen-reader, contrast, zoom/reflow, and interaction testing rather than claim certification. Source: W3C WCAG 2.2 and WAI overview.
- WCAG 2.2 adds AA criteria including focus not obscured, dragging alternatives, and minimum target size. Applicability: sticky navigation, lightboxes, app controls, and mobile targets require explicit review even if older accessibility checklists would miss them. Source: W3C “What's New in WCAG 2.2.”
- Google Search documents that descriptive, distinct title elements, visible main headings, anchor text, and `og:title` can influence search title links, while site link structure affects sitelink discovery. Applicability: repeated generic titles, inconsistent primary headings, and incomplete navigation weaken both search comprehension and shared-link context. Source: Google Search Central title-link and sitelink documentation.
- Core user experience should be judged on observable loading, interaction, and layout stability rather than file size alone. Applicability: image-heavy art/travel experiences, CDN use, Google Drive media, and browser-fetched content deserve representative measurement and graceful-failure inspection before priority claims are finalized.

### Edge Cases

- The deployment may contain stale GitHub Pages artifacts or cached resources not present in the authoritative workspace.
- Publicly reachable but intentionally unlisted utility/editor pages may be operational tools rather than content intended for primary navigation.
- Mixed English/Bulgarian content may be intentional by audience; page-level and passage-level language treatment must be evaluated without presuming monolingual design.
- Third-party content may load slowly, reject embedding, require cookies, rate-limit requests, or be inaccessible to automated review.
- Responsive, keyboard, focus, zoom, reduced-motion, and screen-reader behavior cannot be inferred completely from static source.
- A direct email route can be both valuable professionally and undesirable for spam/privacy; alternatives must preserve contactability.
- Old employment and certification wording may be accurate historical detail rather than stale content, so the critique should flag verification needs without inventing corrections.
- Image dimensions and byte weight alone do not establish real-world performance; representative live measurements are required.
- A strategic Markdown artifact can describe source evidence by filename but must not drift into file-by-file work instructions.
