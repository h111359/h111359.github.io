## Overview

- Request ID: R-20260720-0731
- Request title: Replace media deletion with visibility toggles

### Background

The owner uses the standalone China gallery data editor and wants to preserve control over public media visibility, reduce the friction of selecting the local `data/` folder, and see external changes to the selected event file without reopening it.

### Scope

The request covers `202507_china/data_editor.html`, the serialized `window.GALLERY_ITEMS` media schema, and `202507_china/app.js`. It adds media visibility alongside existing deletion, remembered folder/event restoration, manual refresh, and five-second foreground polling of the selected event file.

### Out of scope

Changing text-block visibility, removing any existing Delete action, refreshing `events.js` during polling, changing registry deletion, introducing server/build dependencies, and bulk-migrating existing event data files are excluded.

## Input Interpretation

### User Original Inputs

- “Instead of delete button next to each image or video, make a flag if the image should be visible - toggle visible or not”
- “Make the file `202507_china/data_editor.html` automatically to open the folder `./202507_china/data` when loaded”
- “Add refresh button + automatic refresh of the currently loaded file each 5 seconds”
- Q001–Q013 answers select `visible` with a visible-by-default legacy fallback; media-only visibility; retained Delete actions for all items plus an additional native “Visible” checkbox; remembered directory-handle and last-event restoration; confirmation before manual dirty refresh; deferred automatic replacement while dirty; selected-event-only refresh; foreground-only polling with an immediate return check; last-good-state error handling; full-text change comparison; and an explicit “No items” all-hidden state.

### AI interpretation

The editor must add a native “Visible” checkbox to image and video cards while retaining the existing Delete action for every item type. Media visibility is persisted as `visible`, with a missing property interpreted as visible; text blocks neither serialize nor honor the flag. The public gallery must omit media with `visible: false` and show “No items” when filtering leaves nothing to render. The editor must restore a previously authorized local `data/` directory handle and last selected event on startup, while retaining the Open folder action when restoration is unavailable. It must expose a manual reload that confirms before discarding dirty edits and poll the selected event file every five seconds while the tab is visible, comparing full source text, deferring external replacement while dirty, retaining the last valid state on read or parse errors, and checking immediately when the tab becomes visible again.

## Research Results

### Workspace pattern-scan

- `202507_china/data_editor.html` is a dependency-free single-file editor. Media and text cards share `makeCardHeader()`, whose Delete action removes either type, so the media checkbox must be added without disturbing that shared action.
- Media serialization currently writes only `name`, `preview`, and `desc`; media creation, DOM synchronization, parsing, and `buildOutput()` must consistently default and preserve `visible`.
- `loadEvent()` already reacquires the selected file handle and resets editor state, making it the natural shared parsing path after conflict checks and text comparison are separated from state replacement.
- `openFolder()` calls `showDirectoryPicker({ mode: 'readwrite' })` and loads `events.js`; persistent restoration requires IndexedDB because localStorage cannot store a `FileSystemDirectoryHandle`.
- `202507_china/app.js` normalizes media and text together and currently rejects only a raw empty array. It must filter only media whose `visible` value is exactly `false`, then distinguish an all-hidden normalized result from a load failure.
- The repository has no package manager or automated test harness, so verification must remain dependency-free and include a supported-Chromium manual acceptance pass.

### Industry knowledge

- `showDirectoryPicker()` requires transient user activation. Persisting a previously selected handle and querying existing permission is the viable startup-restoration model; an unavailable or unauthorized handle must fall back to the existing button.
- File System Access refreshes should call `getFile()` again and then read `text()` because a prior `File` object is only a snapshot. Full-text comparison avoids re-rendering unchanged files.
- Browser timers may be throttled in hidden tabs. The Page Visibility API supports pausing the interval while hidden and checking once immediately after visibility returns.
- Backward-compatible schema evolution uses an optional field with a defined missing-value default. Treating only `visible: false` as hidden keeps all legacy media visible without migration.

### AI Agent critique

- The Q003 answer intentionally changes the original “instead of delete” wording: Delete remains for all elements and visibility becomes an additional media control; the answered Q-block is the later, more specific direction.
- Silent opening of `./202507_china/data` is impossible under browser permission rules; restoration works only for a handle previously chosen and still authorized for the same origin.
- `data_editor.html` evaluates trusted local event JavaScript with `new Function`; malformed external edits can throw and must not replace the last valid editor state.
- Dirty state is boolean-only. A deferred external change therefore needs a pending-refresh marker or equivalent retry after Save makes the editor clean.
- Poll callbacks can overlap when filesystem reads exceed five seconds; the implementation needs an in-flight guard.
- Persisted selection may point to an event removed from `events.js`; restoration must leave selection empty or fall back without treating it as a valid current handle.
- `app.js` currently has no dedicated empty normalized-result renderer, so “No items” must be a successful UI state rather than the red load-error path.
- `document.execCommand()` is deprecated, but that pre-existing rich-text concern is outside this request.
- Context currently describes full item CRUD and a required three-field media schema; the former remains accurate because Delete is retained, while the latter needs an optional visibility qualification.
- The duplicate registry event and empty event title recorded in context remain unrelated and must not be changed.

### Edge Cases

- Legacy media has no `visible` field; malformed non-boolean values must remain visible under the exact-false rule.
- Text blocks must remain rendered even if they happen to contain a `visible` property.
- An event containing only hidden media must show “No items”; an event with hidden media and text must still show its text.
- A visibility click followed by Delete or reorder must preserve unsynchronized field edits.
- Manual refresh while dirty may be cancelled; acceptance leaves state untouched.
- External change while dirty is detected but not applied until the editor becomes clean.
- The selected file may be unchanged, malformed, empty, renamed, deleted, or temporarily unreadable.
- Startup handle permission may be granted, promptable, denied, revoked, unsupported, or stored for another origin.
- The stored last event may no longer exist in the restored registry.
- A poll may already be in flight when another timer or manual refresh fires.
- A hidden tab may remain suspended for longer than one interval and must perform one immediate check on return.
- Refresh before event selection must be disabled or safely do nothing.

## Proposed Solution

### High-Level Concept

Add a backward-compatible media-only visibility flag without removing deletion, and layer safe persisted folder/event restoration plus change-aware refresh behavior onto the existing local editor. Filter hidden media at gallery normalization and render a clear successful empty state.

### Execution Steps

#### Task 1: Add editor visibility state and controls

- `202507_china/data_editor.html`: Default media `visible` to true, render and synchronize a native labeled checkbox alongside the retained Delete button, and serialize `visible` for media only.
  - The editor and gallery must both interpret only `visible: false` as hidden.

#### Task 2: Restore folder access and selected event

- `202507_china/data_editor.html`: Persist directory handles in IndexedDB and the last event identity, restore them when existing permission permits, and retain the Open folder fallback when restoration is unavailable.

#### Task 3: Add safe manual and automatic refresh

- `202507_china/data_editor.html`: Add selected-file refresh UI, confirm manual dirty-state discard, compare fresh full source text during non-overlapping five-second foreground polls, defer changed-source replacement while dirty, retry after the editor becomes clean, and preserve the last valid view with non-blocking status on failures.

#### Task 4: Filter public media

- `202507_china/app.js`: Exclude image/video records with `visible: false`, never apply the rule to text blocks, and render “No items” when successful normalization produces no displayable entries.

#### Task 5: Verify behavior

- `rg`: Confirm `visible`, restoration, refresh, polling, dirty-state, and empty-state logic appears in the intended editor/gallery paths; expected output lists consistent matches in both target files.
- `git diff --check`: Confirm modified files have no whitespace errors; expected output is empty with exit code zero.
- `202507_china/data_editor.html`: Manually verify visibility persistence, retained deletion, folder/event restoration, manual conflict confirmation, five-second external refresh, dirty deferral, hidden-tab pause, and last-good error recovery in Chromium.

#### Task 6: Update product context

- `.aib_memory/context.md`: Record optional media visibility and the editor’s restoration/refresh behavior while preserving the static architecture, required base media fields, trusted rich-text rendering, offline editing, and existing Delete capability.

### Expected Outcome

The editor retains deletion for every item and additionally lets the owner hide media non-destructively; the gallery omits explicitly hidden media while preserving legacy and text entries; and authorized startup restoration plus safe manual/foreground automatic refresh keeps the selected event synchronized without silently discarding edits.
