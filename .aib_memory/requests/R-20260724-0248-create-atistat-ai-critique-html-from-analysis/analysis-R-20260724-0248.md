## Overview

### Background

The `atistat/` folder in this repository is a local static mirror of the production WordPress site `atistat.com`. A prior request (R-20260723-2300) produced a comprehensive technical, SEO, content, accessibility, and visual audit of the site. The site owner requires a public-facing page — `atistat/ai-critique.html` — that presents the evaluation findings and concrete improvement actions in Bulgarian, styled to match the existing ATISTAT brand identity. The page will serve as a handoff artifact from the AI audit to the human site owner and their development team.

### Scope

- Create `atistat/ai-critique.html` as a standalone static HTML page.
- Page content: Executive Summary (1-minute Bulgarian read), Pros/Cons (5-minute Bulgarian read), and Improvement Suggestions (Bulgarian labels, English AI prompts per task).
- Visual identity: same header, footer, CSS, typography, and colour scheme as `atistat/index.html`.
- Source material: audit findings from `analysis-R-20260723-2300.md` (R-20260723-2300 archived document).
- Optionally: link the new page from the existing site navigation (decision pending — see Decision Points).

### Out of scope

- Implementing any of the listed improvement suggestions on existing atistat pages.
- Creating an English-language version of the critique page.
- Modifying files outside the `atistat/` folder.
- Server-side configuration changes (HTTPS redirect, HTTP response headers, compression, caching).
- WordPress admin, plugin, or theme-infrastructure changes not representable as static file edits within `atistat/`.

### Industry Knowledge

1. **Non-technical audit report design (Nielsen Norman Group — Communicating UX Research):** Effective quality reports for non-technical stakeholders require plain-language executive summaries leading with the overall verdict, followed by structured pros/cons, and ending with prioritised, actionable recommendations. Scoring tables are useful for technical audiences but can feel abstract without narrative context; best practice is to pair scores with one-sentence plain-language interpretations.

2. **Static HTML pages mirroring WordPress themes (WordPress Codex — Theme Anatomy; Smashing Magazine — Static Site Generation):** Creating a standalone HTML page that visually matches a WordPress theme requires: referencing the same external CSS file, copying the header and footer HTML verbatim (or minimally adapted), and reproducing the WordPress body classes that the CSS targets. Since the `atistat/` folder is a static mirror, no PHP template inheritance is available — the header and footer HTML must be copied directly into the new file.

3. **AI copilot prompt engineering for code changes (GitHub Copilot Documentation; OpenAI prompt engineering guide):** Effective GitHub Copilot chat prompts for file modification tasks are self-contained, specify the exact target file path, describe the problem being fixed, state the expected outcome in observable terms, and include any specific constraints. Prompts phrased as "In file X, change Y to Z because W" yield more reliable, reproducible results than open-ended instructions.

### AI Agent Critique

- **input.md goal clause ambiguity:** The goal says "based on `.aib_memory/requests/.../analysis-R-20260723-2300.md`" yet also prohibits mentioning this path in page texts. This is internally consistent (source is confidential; page content is derived, not quoted), but the implementer must keep the boundary clear when writing page content.
- **analysis-R-20260723-2300.md — server-side task mismatch:** Several top-priority issues in the audit (HTTPS redirect, HSTS, security headers, compression, caching) require Apache or server-level configuration changes. These cannot be represented as `atistat/` file edits. The improvement-task list must omit or annotate these as server-side concerns outside file scope.
- **analysis-R-20260723-2300.md — project pages:** The six project pages in `atistat/proekti/` were rated as most severely deficient (thin content). Including improvement tasks for them is within scope, but the prompts must acknowledge that substantive content (project facts, images, case-study text) requires input from the site owner that AI cannot generate from available data.
- **atistat/index.html — duplicate SVG IDs:** Both header and footer contain SVG elements with the same IDs (`svg1`, `defs1`, etc.). This violates HTML uniqueness requirements and can cause assistive-technology confusion. An improvement task to fix this is within scope.
- **atistat/index.html — `.at-fade` resilience defect:** Content is hidden by default via `.at-fade` CSS; JavaScript must run to reveal it. This is a confirmed resilience defect. An improvement task exists; the fix involves a CSS and/or JS change within `atistat/wp-content/themes/atistat/assets/css/main.css` and/or `main.js`.
- **context.md Issues section:** Contains the `aib-analyze.md` self-critique note about prompt duplication. Not relevant to this request.
- **atistat/index.html — hero image lazy loading:** The visible hero image uses `loading="lazy"`, which is the confirmed cause of the 3.86 s mobile LCP. Fix: change to `loading="eager"` and add `fetchpriority="high"`. This is a one-attribute change within `atistat/index.html`.
- **atistat/index.html — empty preconnect:** `<link rel="preconnect" href="">` at line 6 is empty and provides no benefit. A quick fix within `atistat/index.html`.

### Edge Cases

- **Nav anchor dead links:** The header nav on `atistat/index.html` uses same-page anchors (`#about`, `#opit`, `#services`, `#contacts`). On `atistat/ai-critique.html`, these sections do not exist. Clicking nav links will scroll to the page top or do nothing — not an error, but a UX gap. Resolution requires a decision on nav anchor behaviour (see Decision Point: nav-anchor-behavior).
- **Language switch on critique page:** The existing BG/EN toggle links to `index-en.html`. On the critique page, there is no English equivalent. If the toggle is preserved, the EN link becomes contextually misleading. Resolution requires a decision on language-switch treatment (see Decision Point: language-switch).
- **noindex vs. public accessibility:** The page is inside `atistat/`, which maps to a live domain. If the page is indexed, a search for "ATISTAT critique" could surface it publicly, which may not align with the owner's intent. Resolution requires a decision on noindex (see Decision Point: noindex-meta).
- **CSS relative path:** `wp-content/themes/atistat/assets/css/main.css` is referenced as a relative path from `atistat/index.html`. Since `ai-critique.html` is also in the `atistat/` root, the same relative path is valid and requires no adjustment.
- **Font path:** Font files are referenced as `../fonts/bookman-cyr.woff2` relative to the CSS file (`atistat/wp-content/themes/atistat/assets/css/main.css`), which resolves to `atistat/wp-content/themes/atistat/assets/fonts/`. This path is independent of the HTML file location — correct for both `index.html` and `ai-critique.html`.
- **JS script path:** `atistat/wp-content/themes/atistat/assets/js/main.js` is referenced relatively. Same validation: resolves correctly from both pages.
- **Improvement prompt references:** Prompts for tasks that target CSS or JS files must reference the full relative path as seen from the repository root (e.g., `atistat/wp-content/themes/atistat/assets/css/main.css`) to be executable by GitHub Copilot without ambiguity.

## Proposed Solution

### High-Level Concept

Create `atistat/ai-critique.html` as a single standalone HTML file that reuses the ATISTAT theme CSS and mirrors the header/footer HTML from `atistat/index.html`, presenting the audit findings in Bulgarian across three thematically styled sections with an embedded set of English AI prompts for improvement tasks.

### Execution Steps

#### Task 1: Create atistat/ai-critique.html

- `atistat/ai-critique.html`: Create new file with complete HTML5 structure including DOCTYPE, `<html lang="bg">`, `<head>` with charset, viewport meta, title, CSS link (`wp-content/themes/atistat/assets/css/main.css`), and `<meta name="robots" content="noindex, nofollow">` to prevent search engine indexing.
- `atistat/ai-critique.html`: Add body with WordPress body classes matching index.html (`wp-theme-atistat`) and the ATISTAT header block (skip link, `<header class="at-header">` with logo SVG, mobile nav toggle, `<nav class="at-nav">` containing only a single "← Обратно към АТИСТАТ" link pointing to `index.html`).
  - No same-page anchor links (#about, #opit, etc.) and no language switch (`<div class="at-lang">`) on this page.
- `atistat/ai-critique.html`: Add `<main id="main" class="at-main">` with three `<section class="at-section">` blocks:
  - Section 1 `#executive-summary`: Bulgarian Executive Summary (~250–300 words) covering the overall qualitative verdict, top 3 strengths, and the 3 most urgent weaknesses. No numeric scores included.
  - Section 2 `#pros-cons`: Bulgarian Pros/Cons in two sub-sections (`<h3>` Предимства / Недостатъци), covering qualitative analysis across technical, SEO, content, visual, UX, accessibility, and mobile dimensions. No scoring table or numeric scores — qualitative narrative only.
  - Section 3 `#improvements`: Bulgarian-labelled improvement task list covering Critical and High priority issues only (~8 tasks within `atistat/` file scope, excluding server-side issues). Each task has: a Bulgarian heading with an inline priority tag (e.g., [Критично], [Важно]), a Bulgarian description, and an English AIB brain prompt block (using `## Input` / `## Goal` structure) in a `<pre><code>` or `<details>` element.
- `atistat/ai-critique.html`: Add `<footer class="at-footer">` block matching index.html footer (logo SVG, copyright notice, back-to-top link).
- `atistat/ai-critique.html`: Add `<script>` reference to `wp-content/themes/atistat/assets/js/main.js` before closing `</body>` to enable nav toggle and fade-in animations.
