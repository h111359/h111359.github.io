## Overview

### Background

The `atistat/` folder in this repository is a local static mirror of the production WordPress site `atistat.com`. The page will serve as a handoff artifact from the AI audit to the human site owner and their development team.

- Page content: Executive Summary (1-minute Bulgarian read), Pros/Cons (5-minute Bulgarian read), and Improvement Suggestions (Bulgarian labels, English AI prompts per task).

- Source material: audit findings from `analysis-R-20260723-2300.md` (R-20260723-2300 archived document).
- Optionally: link the new page from the existing site navigation (decision pending — see Decision Points).

### Out of scope

- Creating an English-language version of the critique page.
- Modifying files outside the `atistat/` folder.
- Server-side configuration changes (HTTPS redirect, HTTP response headers, compression, caching).
- WordPress admin, plugin, or theme-infrastructure changes not representable as static file edits within `atistat/`.

### Industry Knowledge

1. **Non-technical audit report design (Nielsen Norman Group — Communicating UX Research):** Effective quality reports for non-technical stakeholders require plain-language executive summaries leading with the overall verdict, followed by structured pros/cons, and ending with prioritised, actionable recommendations. Scoring tables are useful for technical audiences but can feel abstract without narrative context; best practice is to pair scores with one-sentence plain-language interpretations.

2. **Static HTML pages mirroring WordPress themes (WordPress Codex — Theme Anatomy; Smashing Magazine — Static Site Generation):** Creating a standalone HTML page that visually matches a WordPress theme requires: referencing the same external CSS file, copying the header and footer HTML verbatim (or minimally adapted), and reproducing the WordPress body classes that the CSS targets. Since the `atistat/` folder is a static mirror, no PHP template inheritance is available — the header and footer HTML must be copied directly into the new file.

3. **AI copilot prompt engineering for code changes (GitHub Copilot Documentation; OpenAI prompt engineering guide):** Effective GitHub Copilot chat prompts for file modification tasks are self-contained, specify the exact target file path, describe the problem being fixed, state the expected outcome in observable terms, and include any specific constraints. Prompts phrased as "In file X, change Y to Z because W" yield more reliable, reproducible results than open-ended instructions.

## Decision Register

### Decision Points

#### Decision Point: nav-integration

- **Tag:** ask
- **Applies to:** Task 2 and header nav construction in Task 1.
- **Why alternatives exist:** The input specifies creating the page but does not specify whether it should appear in the main site navigation. Adding a nav link requires modifying `atistat/index.html` and `atistat/index-en.html`, which would affect every visitor to the live atistat.com site. Not adding a link keeps the page accessible only by direct URL.

Alternatives:

- **Alternative A — Add nav link to all atistat pages:** Insert a nav item in `atistat/index.html` and `atistat/index-en.html` header nav. Every site visitor sees a link to the critique. Benefit: discoverable. Drawback: publicly exposed evaluation may not align with owner intent; modifies two existing pages.
- **Alternative B — Standalone page, no nav link:** `ai-critique.html` is accessible by direct URL only. No changes to existing pages. Benefit: keeps the page internal/private. Drawback: requires manual URL sharing.
- **Alternative C — Footer link only:** Add a small, discrete link in the footer of `atistat/index.html` and `atistat/index-en.html`. Less prominent than full nav entry.

#### Decision Point: ai-prompt-format

- **Tag:** ask
- **Applies to:** Section 3 (Improvement Suggestions) prompt block content in Task 1.
- **Why alternatives exist:** The input says "exact prompt for the AI copilot to execute" but does not specify the format. The workspace contains both AIB brain prompts (`.aib_brain/prompts/`) and plain GitHub Copilot chat prompts. These produce different prompt structures and expected execution flows.

Alternatives:

- **Alternative A — GitHub Copilot chat prompts (recommended):** Plain natural-language prompts prefixed with the target file, change description, and expected outcome. Executable by pasting into GitHub Copilot Chat. No special tool chain required.
- **Alternative B — AIB brain prompts:** Structured prompts using `## Input` / `## Goal` format triggering the aib-analyze.md workflow. Requires the `.aib_brain/` workflow to be active.
- **Alternative C — Generic task descriptions:** Non-tool-specific plain-English improvement instructions without any particular format. Easy to read but not directly executable.

#### Decision Point: task-count-and-scope

- **Tag:** ask
- **Applies to:** Section 3 task list size and content in Task 1.
- **Why alternatives exist:** The audit identified 24 distinct issues. The input does not specify how many improvement tasks to include. Including all 24 creates a comprehensive but long list; filtering by priority produces a more actionable document.

Alternatives:

- **Alternative A — All actionable within atistat/ (estimated 15 tasks):** Include every issue fixable via `atistat/` file edits; excludes only server-side issues. Comprehensive but longer.
- **Alternative B — Critical + High priority issues only (estimated 10 tasks):** Include only issues rated Critical or High in the audit. Focused and more immediately actionable.
- **Alternative C — Top 5 most impactful issues:** Very concise; covers only the five issues with the greatest expected improvement to site quality.

#### Decision Point: noindex-meta

- **Tag:** ask
- **Applies to:** `<head>` section of `atistat/ai-critique.html` in Task 1.
- **Why alternatives exist:** The page is inside the `atistat/` folder which maps to `atistat.com`. If the page is publicly indexed by search engines, an evaluation of ATISTAT's site weaknesses could appear in search results, which may not be the owner's intent.

Alternatives:

- **Alternative A — Add `<meta name="robots" content="noindex, nofollow">`:** Page is excluded from search engine indices. Benefit: keeps evaluation private from public search. Drawback: page is not discoverable via search.
- **Alternative B — No special robots meta:** Page is indexable. Benefit: consistent with other site pages. Drawback: public visibility of internal evaluation content.

#### Decision Point: nav-anchor-behavior

- **Tag:** ask
- **Applies to:** Header nav link targets in `atistat/ai-critique.html` in Task 1.
- **Why alternatives exist:** The header nav links (`#about`, `#opit`, `#services`, `#contacts`) are same-page anchors designed for `index.html`. On the critique page, these sections do not exist, creating dead anchor links.

Alternatives:

- **Alternative A — Prefix anchors with `index.html`:** Change nav hrefs to `index.html#about`, `index.html#opit`, etc. Clicking any nav item navigates to the main homepage at the relevant section. Benefit: fully functional nav. Drawback: moves user away from the critique page.
- **Alternative B — Keep anchors as-is:** No change; clicking nav items scrolls to page top. Benefit: header HTML identical to index.html (easier maintenance). Drawback: misleading nav behaviour.
- **Alternative C — Simplified nav for critique page:** Replace section anchors with a single "← Обратно към АТИСТАТ" link pointing to `index.html`. Benefit: clear navigation intent. Drawback: diverges from brand nav pattern.

#### Decision Point: language-switch

- **Tag:** ask
- **Applies to:** Section 2 (Pros/Cons) content in Task 1.
- **Why alternatives exist:** The audit produced a detailed 12-category scoring table and a 52/100 weighted overall score. Including scores adds specificity and credibility to the report; omitting them keeps the presentation non-technical and avoids over-emphasis on numeric ratings.

Alternatives:

#### Decision Point: task-priority-labels

- **Tag:** ask
- **Applies to:** Section 3 task list presentation in Task 1.
- **Why alternatives exist:** The audit assigns each issue a priority level (Critical, High, Medium, Low). Surfacing priorities in the improvement task list helps the owner triage effort; omitting them keeps the list focused on what needs to change rather than ranking.

Alternatives:

- **Alternative A — Label each task with its priority level (Critical/High/Medium):** Inline tag per task (e.g., `[Critical]`, `[High]`). Benefit: owner can prioritise quickly. Drawback: longer task entries; Critical/High framing may feel alarmist.
- **Alternative B — Group tasks by priority level (three sections):** Render three `<h3>` sub-sections: Критично → Важно → Средно. Benefit: visually clear priority tiers. Drawback: breaks single list flow.
- **Alternative C — No priority labels, ordered by impact:** Tasks ordered from highest to lowest impact without explicit labels. Benefit: streamlined; reads as a recommended action sequence. Drawback: priority intent is implicit only.
