## Overview

### Background

The `202507_china/` gallery stores all event data as JavaScript files under `202507_china/data/`. Each file defines `window.GALLERY_ITEMS`, an array of photo entries (`name`, `preview`, `desc`) and text block entries (`type`, `title`, `body`, `desc`). An existing partial editor (`comments_index.html` + `comments_app.js`) supports upload/download of single event files and exposes limited field editing: only `desc` for media items, and `type`, `title`, `body`, `desc` for text blocks. The developer requires a fully-featured local editor with complete field coverage for all item attributes, rich text formatting in text fields, item management (add, delete, reorder), and multi-event navigation — all running purely in a local browser with no server.

### Scope

- New or enhanced HTML editor targeting all files in `202507_china/data/`
- Full field editing for photo/video entries: `name`, `preview`, `desc`, and any new fields (e.g., `size`)
- Full field editing for text block entries: `type`, `title`, `body`, `desc`
- Rich text formatting support in text fields (bold, colors, newlines)
- Item creation, deletion, and reordering within an event
- Event file navigation across all 47 events
- All functionality operating locally in a browser via `file://` protocol without server-side code

### Out of scope

- Publishing or deploying changes to GitHub Pages
- Server-side components or build tools
- Editing non-gallery files (root HTML pages, CSS, or JS files outside `202507_china/data/`)
- Uploading or managing media files on Google Drive
- Editing the gallery viewer (`app.js`, `style.css`) except where strictly required by the chosen rich text format approach

### AI interpretation

The developer requests a browser-based, fully local editor targeting the per-event JavaScript data files in `202507_china/data/`. The editor must provide complete CRUD control over `window.GALLERY_ITEMS` arrays in each event file: reading file content, rendering all items in an editable UI, allowing creation, editing, deletion, and reordering of items, and writing modified content back to disk.

The editor must expose all currently defined fields per item type:

- **Media entries** (image/video): `name` (filename reference), `preview` (Google Drive URL), `desc` (caption/description), and a new `size` field if the developer confirms its intended meaning
- **Text block entries** (`type: text`): `type`, `title`, `body`, `desc`

The phrase "text formatting (new lines, bold, coloring, etc.)" indicates that `desc` and `body` text fields must support inline formatting. The current gallery viewer (`app.js`) escapes all text content via `esc()` and only converts `\n` to `<br>`. Supporting bold and color would require either storing HTML markup in the string fields and updating the gallery viewer, or adopting a Markdown or custom syntax. The preferred approach is unresolved and must be clarified.

"Loaded in my local browser" means the editor must function as a pure static file via the `file://` protocol with no server. File I/O must use browser-native APIs (File API or File System Access API). The project convention (from `context.md`) mandates pure static HTML, CSS, and vanilla JavaScript with no build toolchain.

The project has an existing partial editor (`comments_index.html` / `comments_app.js`). Whether the new editor replaces, extends, or coexists with it is unresolved and must be clarified.

### Industry knowledge

**1. File System Access API (WICG/W3C, Chrome 86+):**
The File System Access API (`showOpenFilePicker`, `showSaveFilePicker`, `showDirectoryPicker`) allows web pages opened via `file://` or `https://` to request persistent read/write access to files on disk. Using `showDirectoryPicker`, the editor can read all `.js` files in the `data/` folder and write back changes in place without any upload/download cycle. Key limitation: not supported in Firefox (status: `under consideration` in Mozilla's web-compat tracker, not shipped as of 2025). Applicability: highly relevant when the developer uses Chrome or Edge; requires graceful fallback for Firefox users.

**2. Rich text editing in vanilla JavaScript (WHATWG / MDN standards):**
The `contentEditable` attribute combined with custom keyboard event handlers and a formatting toolbar allows WYSIWYG editing without external libraries. `document.execCommand()` (bold, foreColor, etc.) is deprecated in the WHATWG living standard but continues to work in all major browsers. The preferred modern approach is a `contentEditable` element with `document.execCommand()` for simple formatting needs, accepting the deprecation notice for a local-only tool. For Markdown-based editing, a plain `<textarea>` with a live preview div (rendered via a small Markdown parser) provides a no-library alternative. Applicability: both approaches are viable for a local-only tool with no CDN dependency requirement.

**3. Local-first editing patterns (Decap CMS, Publii, static-site tooling):**
Projects like Decap CMS (formerly Netlify CMS) and Publii follow a data-model-first editing pattern: parse file → build internal model → render editable widgets → serialize model back on save. The consistent recommendation is strict separation of the data model (JS objects) from the UI rendering and from the JS serialization layer. The current `comments_app.js` partially implements this pattern but uses the source textarea as both model and output, which creates tight coupling. Applicability: a clean data model layer would improve reliability of the new editor.

**4. JavaScript object serialization for static-site data files:**
Reconstructing `window.X = [...]` output from an edited model requires a dedicated serializer that preserves property order, quoting style, and indentation. The existing `buildOutput()` function in `comments_app.js` handles the current limited schema correctly using template strings and a `jsStringEscape()` helper. Extending this to new fields requires updating `buildOutput()` and ensuring the `jsStringEscape()` function handles all characters that may appear in user-entered text (including Unicode, HTML entities if HTML markup is used, and control characters).

**5. XSS considerations for stored HTML in data files (OWASP Top 10 — A03: Injection):**
If formatted HTML is stored in `desc` or `body` fields, the gallery viewer rendering those fields with `innerHTML` introduces a stored XSS risk within the context of the gallery page. For a local-only, single-user tool with no public-facing deployment, the practical risk is low. However, the current `esc()` function in `app.js` would need to be replaced with a safe HTML rendering approach (either trusted-content bypass with no sanitizer, or a DOMPurify-style sanitizer). Since `comments_app.js` already uses `escapeHtml()` for display and `jsStringEscape()` for output, the serialization layer is safe; the risk is in the gallery viewer consumption.

### AI Agent critique

- `202507_china/data/events.js`: Contains a duplicate entry for `event-20250724-tudja`; two identical `{ slug: "event-20250724-tudja", title: "20250724_08 Племето Туджа от народа Ба", data: "..." }` entries. The gallery dropdown will show this event twice. Likely a data entry error.

- `202507_china/data/events.js`: Event `event-20250726_08-3gorges` has an empty `title: ""`. The gallery dropdown will show a blank label for this event.

- `202507_china/comments_app.js` — `tryParseItems()`: Uses a broad regex `([{\,]\s*)([a-zA-Z_$][\w$]*)\s*:` to add quotes to unquoted property names. This will fail for any property name that starts mid-line without a preceding `{` or `,`, e.g., on indented continuation lines. Also strips trailing commas in `arrStrClean` generation but `arrStrQuoted` trailing-comma stripping via `/([\s]*[}\]])` is overly permissive.

- `202507_china/comments_app.js` — `tryParseItems()`: The `_checked` flag mapping uses `jsonLines[idx]?.checked`, but `idx` is the loop index in `raw.map((it, idx) => ...)` from the parsed JSON array. If any `jsonLines` entries were filtered out (e.g., due to empty lines being skipped), the indices will be misaligned, resulting in incorrectly marked checked/unchecked states.

- `202507_china/comments_app.js` — `jsStringEscape()`: Does not handle Unicode surrogate pairs, null bytes (`\0`), or non-breaking spaces that may appear in imported text from external sources (e.g., copied from a web page). These could produce malformed output JS.

- `202507_china/app.js` — `makeTextCard()`: Uses `esc(item.body).replace(/\n/g,'<br>')`. This approach is incompatible with HTML-formatted content: if `body` stores `<b>bold</b>`, `esc()` will output `&lt;b&gt;bold&lt;/b&gt;`, rendering literally rather than as bold text.

- `202507_china/app.js` — `makeImageCard()`: The `img.onerror` fallback retries with `thumb(item.id, 1600)`, but if `item.id` was already derived from the same URL and is valid, the retry is likely to succeed; however if `extractId()` failed and returned `null`, neither URL is valid and the error is silently swallowed.

- `202507_china/comments_app.js`: The `render()` function calls `regenerate(items)` at the end, which overwrites the source textarea with regenerated JS. If the user has partially typed unsaved content in the textarea before clicking "Upload", the upload handler calls `src.value = content` and then `autoLoadItems()` which calls `render()` → `regenerate()`, which is correct. However, if the user edits the textarea directly (typing) and those edits trigger `autoLoadItems()`, the parse may fail on invalid intermediate states, clearing the rendered grid.

### Edge Cases

- **Empty event files**: `window.GALLERY_ITEMS = [];` — editor must handle gracefully without crashing.
- **Commented-out items at parse time**: Items with `//` prefix must be parsed, displayed with a "disabled" visual indicator, and preserved in output.
- **Large events**: Some events have 50+ items (e.g., `event-20250721_05-lui-vuiton.js` has ~60 items); the editor's rendering must remain performant.
- **Unsaved changes on event switch**: If the user switches to a different event without saving, changes could be lost; a confirmation prompt or auto-save is required.
- **HTML injection risk**: If HTML markup is stored in `body`/`desc` and rendered via `innerHTML` in the gallery viewer without sanitization, stored XSS is possible in the gallery page context.
- **Invalid Google Drive URLs**: A `preview` value not matching `https://drive.google.com/file/d/<id>/preview` causes `extractId()` in `app.js` to return `null`, making the item invisible in the gallery.
- **File System API browser support**: The File System Access API is unavailable in Firefox; the editor must either detect this and fall back to File API, or document the requirement.
- **Encoding edge cases**: Cyrillic, emoji, and other non-ASCII characters are already present in data files (confirmed in event titles and `desc` fields); the serializer must output these without escaping them to `\uXXXX`.
- **Missing optional fields in existing files**: If a new `size` field is introduced, all existing event files lack it; the editor must gracefully default missing optional fields.
- **Concurrent external edits**: If a file is edited outside the editor while it is open, the editor has no mechanism to detect the external change.
- **Template file**: `event-20250724-template.js` contains placeholder items; the editor should load it without special-casing but the user should be aware it is a template.

#### Decision Point: file-access-mechanism

- Tag: ask
- Rationale: The editor must load and save `202507_china/data/*.js` files from the local filesystem. Two browser API approaches are viable. No workspace document specifies a preference, and the user's browser is unknown.

Alternatives:

- Option A: File API — `<input type="file">` for upload and `Blob` download via anchor element; works in all browsers including Firefox; requires manual upload/download per file per session
- Option B: File System Access API — `showDirectoryPicker()` for folder access; direct in-place file read/write; no manual upload/download; Chromium-based browsers only (Chrome 86+, Edge 86+; not Firefox)
- Option C: Hybrid — attempt File System Access API on load; if unavailable, fall back to File API automatically

- Rationale: It is unclear whether the user wants to edit multiple events simultaneously in one browser session, or work on one event at a time. The current editor handles one file per session.

Alternatives:

- Option A: One event at a time — switching events prompts to save unsaved changes first
- Option B: Tabbed multi-event — a tab bar within the editor allows multiple events open simultaneously
- Option C: One event at a time — switching discards unsaved changes without prompt (simplest)

#### Decision Point: grid-size-in-editor

- Tag: ask
- Rationale: The size at which photo previews are displayed within the editor affects usability. The current `comments_app.js` shows full-height iframes. For events with 50+ items, large previews require significant scrolling. The developer did not specify a preference.

Alternatives:

- Option A: Small thumbnails (~150px height) — more items visible per viewport
- Option B: Medium previews (~300–400px height) — balance between visibility and density
- Option C: Large previews (~600px+ height) — easy to inspect detail, more scrolling required
- Option D: User-configurable zoom level via a slider or setting

#### Decision Point: CDN-vs-offline

- Tag: ask
- Rationale: The editor is intended for local browser use. If any CDN-loaded library is used (e.g., for rich text or Markdown rendering), the editor will not function without internet access. Offline scenarios (e.g., editing while travelling) are realistic for a China travel photo editor.

Alternatives:

- Option A: Fully offline — no CDN dependencies; all code embedded inline in the HTML file; works on `file://` with no internet connection
- Option B: CDN with graceful degradation — load optional libraries from CDN; detect failure and fall back to reduced functionality when offline
- Option C: CDN required — load libraries from CDN; document that internet connection is required

#### Decision Point: photo-preview-in-editor

- Tag: ask
- Rationale: The current `comments_app.js` embeds a Google Drive iframe for each media item during editing. For events with 50+ items, loading many iframes simultaneously may degrade editor performance. The user may prefer a faster thumbnail approach.

Alternatives:

- Option A: Google Drive iframe preview — embed the Google Drive preview iframe for each media item (same as current `comments_app.js`)
- Option B: Thumbnail only — use the Google Drive thumbnail URL (`https://drive.google.com/thumbnail?id=<id>&sz=w400`) as an `<img>` tag; faster loading than iframes
- Option C: No preview in editor — show only the filename and URL text fields; no visual media preview
