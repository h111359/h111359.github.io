## Goal

Create `202507_china/data_editor.html` — a standalone, fully-offline, browser-based editor for all event JavaScript data files and `events.js` in `202507_china/data/`. The editor provides full CRUD control over `window.GALLERY_ITEMS` items (add, edit all fields, delete, reorder) for both media entries and text block entries, with HTML rich text formatting in `body`/`desc` fields, File System Access API in-place read/write, events.js registry editing, and a dropdown event navigation bar. An accompanying update to `202507_china/app.js` enables the gallery viewer to render stored HTML formatting.

Background: The existing `comments_app.js` partial editor exposes only `desc` editing for media entries and has no event navigation, no item reordering, and no new-item creation beyond text-block insertion. The developer needs a complete editor to efficiently maintain all 47+ event data files without manual JS editing.

Impacted components: `202507_china/data_editor.html` (new file), `202507_china/app.js` (modified), `.aib_memory/context.md` (updated).

## Constraints

- All delivered files must be pure static HTML/CSS/JavaScript; no server-side code, build tools, or package manager.
- The editor must embed all CSS and JavaScript inline in the single `data_editor.html` file; no external script or stylesheet links.
- The editor must run on `file://` without an internet connection; no CDN resources.
- Chromium-based browser (Chrome 86+, Edge 86+) is required for File System Access API (`showDirectoryPicker()`); this restriction must be documented in the editor UI.
- The existing `comments_index.html` and `comments_app.js` files must not be modified or deleted.
- The editor does not create new event JS files from scratch; it edits existing files only.
- Assumption: All event JS files in `202507_china/data/` define `window.GALLERY_ITEMS` at the top level using the established schema (`name`, `preview`, `desc` for media; `type`, `title`, `body`, `desc` for text blocks).
- Assumption: The directory handle obtained from `showDirectoryPicker()` provides read/write access to all files in `202507_china/data/` after the user grants permission.

## Success criteria

- `202507_china/data_editor.html` is a single file with all CSS and JavaScript inline; no external dependencies.
- Opening the file in Chrome/Edge and clicking the folder-open button triggers `showDirectoryPicker()`; selecting the `data/` folder populates the event dropdown from `events.js`.
- Selecting an event in the dropdown loads its `window.GALLERY_ITEMS` and renders all items as editable cards.
- Media entry cards display a Google Drive thumbnail `<img>` at ~300–400px height and expose editable `name`, `preview`, and `desc` (rich text) fields.
- Text block cards expose `type` (read-only), `title` (text input), `body` (rich text), and `desc` (rich text) fields.
- The formatting toolbar applies bold, italic, underline, strikethrough, foreground color, H2, H3, and bulleted list to the focused rich text field.
- Global "Add photo entry" and "Add text block" buttons appear at both the top and bottom of the item list; between-item insert controls appear between each adjacent pair of items.
- Each item card has a Delete button that shows a confirmation dialog before removing the item; Move-up and Move-down buttons swap the item with its neighbour.
- An "Unsaved changes" badge is visible in the editor header after any edit or structural change; the badge disappears after a successful save.
- The Save button writes the serialized `window.GALLERY_ITEMS` back to the original event file in place via File System Access API.
- Switching events while dirty state is active prompts the user to confirm before the current event is discarded.
- Pre-save validation displays a dismissible warning list for items with missing or invalid-format `preview` URLs; save proceeds regardless.
- The events.js registry editor panel lists all entries with editable slug, title, and data-path fields; supports add (append), remove (with confirm), and reorder (up/down); Save writes updated `window.GALLERY_EVENT_INDEX` back to `events.js`.
- In the gallery viewer, opening any event that contains HTML in `body` or `desc` fields renders the formatting correctly rather than displaying literal markup.

### Task 6: Implement serialization and save

#### Intent

Implement `buildOutput()` to serialize `items` to the `window.GALLERY_ITEMS = [...];` format with correct escaping and Unicode preservation, and implement the Save button handler with pre-save validation.

#### Procedure

In `202507_china/data_editor.html` inline script, implement `jsEscape(str)`: a string-escaping helper that replaces `\` with `\\`, `"` with `\"`, and control characters (code points U+0000–U+001F excluding `\n` and `\t`) with their `\uXXXX` equivalents; does not escape any code point above U+007F; does not HTML-encode `<`, `>`, or `&` characters; returns the escaped string.

In `202507_china/data_editor.html` inline script, implement `buildOutput()`: calls `syncItemsFromDOM()` first; builds the output string starting with `window.GALLERY_ITEMS = [\n`; for each item in `items`, outputs a two-space-indented object literal `{ key: "value", ... }` with each property on the same line, using `jsEscape()` on all string values; ends with `];\n`; for media entries includes `name`, `preview`, `desc` properties (omitting a property only if the item was newly created and the field is empty — for existing items that had an empty field, preserve the empty string); for text block entries includes `type`, `title`, `body`, `desc`; returns the built string.

In `202507_china/data_editor.html` inline script, implement `validate()`: iterates `items`; for each item at index `i`, checks: if it is a media entry with an empty `preview`, adds a violation `{ index: i, message: 'Missing preview URL' }`; if it is a media entry with a non-empty `preview` that does not match the regex `^https://drive\.google\.com/file/d/[a-zA-Z0-9_-]+/preview$`, adds a violation `{ index: i, message: 'Invalid Google Drive URL format' }`; returns the violations array.

In `202507_china/data_editor.html` inline script, implement the Save button handler: calls `validate()`; if any violations exist, renders them in a dismissible warning area (list of "Item N: message" lines with a Dismiss/Continue button); awaits the Continue action using a Promise or callback before proceeding; calls `buildOutput()`; calls `currentEventHandle.createWritable()`, writes the output string, closes the writable; sets `dirty = false`; sets the unsaved badge to `hidden`; shows a temporary "Saved ✓" notification in the header for 2 seconds then hides it. Bind this to the Save button `click` event. Disable the Save button while no event is loaded (`currentEventIndex === -1`).
