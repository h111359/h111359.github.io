# ATISTAT Website Audit

Audit date: 23 July 2026  
Request: R-20260723-2300  
Audited source: local mirror in `atistat/`  
Production checks: `https://atistat.com/` where server behavior could not be assessed from static files

## Audit Method and Limits

This audit covers the Bulgarian and English home experiences, the `/opit/` page, all six project routes in the sitemap, language variants stored in the mirror, contact methods, external destinations, responsive behavior, the public 404 response, robots and sitemap files, front-end source, assets, and visible production headers.

Evidence labels used below:

- **Confirmed:** directly present in source, rendered output, or an HTTP response.
- **Measured:** obtained from a single controlled browser or HTTP run on 23 July 2026.
- **Professional judgment:** interpretation based on the confirmed evidence.
- **Manual follow-up:** cannot be established conclusively without broader assistive-technology, field-user, legal, or security testing.

The browser review used Chrome 149 at 1440 × 1000 and a narrow 390-pixel window. A live desktop lab run was unthrottled. The live mobile lab run emulated a narrow mobile device, approximately Fast 4G networking, and 4× CPU slowdown. These are diagnostic lab observations, not field Core Web Vitals and not a substitute for Chrome UX Report data. INP could not be established from a page-load-only test.

The security section is an external configuration review, not a penetration test. Legal and privacy observations are risk flags for qualified review, not legal advice.

## 1. Executive Summary

### Overall assessment

ATISTAT has a professionally conceived visual identity but does not yet meet a complete professional website standard. The homepage is attractive, coherent, and unusually distinctive for a small construction company. The cream-and-green palette, Bookman-style typography, architectural sketches, restrained animation, and chronological project presentation communicate care and domain relevance.

The quality drops sharply outside that presentation layer. Each of the six project pages contains only a project name and a one-line service label. The English project variants are still Bulgarian. There is no persuasive primary call to action, no enquiry form, no substantive case study, no privacy or legal information, and almost no search or social metadata. A narrow-screen rendering defect clips the Bulgarian hero headline. Several interaction patterns are not keyboard-equivalent. Production serves the entire site over unencrypted HTTP instead of redirecting to HTTPS and omits common security headers.

The result feels like a strong visual concept that has been published before its content, conversion, SEO, accessibility, and production-hardening work was finished.

### Apparent purpose and audience

**Confirmed:** The site presents ATISTAT EOOD as a Sofia construction and investment company offering residential construction, renovation and reconstruction, project development, and real-estate investment.

**Professional judgment:** Its primary audiences are property owners, investors, landowners, development partners, and organizations evaluating ATISTAT's experience and credibility. Secondary audiences include prospective suppliers, employees, and people researching the founder or related companies.

### Strongest qualities

1. **Distinctive, industry-relevant design.** The building-sketch motif is memorable and avoids a generic stock-photo construction template.
2. **Clear high-level service framing.** Four services and their processes are described in understandable language.
3. **Useful credibility foundations.** The founder is named, a 26-year professional history is outlined, known projects are listed, related companies are linked, an award claim is included, and a physical address, phone, and domain email are visible.
4. **Lean front-end behavior.** The custom JavaScript is only 3.4 KB, dependency-free, and produced no site-code console error in the observed Chrome run.
5. **Good semantic foundations.** The homepage has a main landmark, labelled navigation, one H1, orderly section headings, meaningful image alternatives, a titled map iframe, a skip link, and a reduced-motion rule.

### Most serious weaknesses

1. **Transport security is not enforced.** `http://atistat.com/` returned `200 OK` with the full site instead of redirecting to HTTPS.
2. **The portfolio is effectively unfinished.** Six indexable project pages expose only a title and one service phrase, with no image, facts, challenge, scope, outcome, or next action.
3. **The site does not guide conversion.** There is no hero CTA, project enquiry CTA, service-level CTA, form, qualification path, or confirmation/follow-up flow.
4. **Search and sharing readiness is very weak.** All inspected pages lack meta descriptions, Open Graph, Twitter Card metadata, and structured data; the home language system lacks hreflang and robust canonicalization.
5. **The English experience is incomplete.** The English homepage is translated, but the six English project pages remain Bulgarian and the sitemap omits a clear English content set.

### Three most urgent improvements

1. Enforce a permanent HTTP-to-HTTPS redirect, then add HSTS and baseline security headers after validation.
2. Repair the mobile headline and accessibility defects, and make content visible by default when JavaScript fails.
3. Turn the six project stubs into linked, bilingual case studies and place a clear “Discuss a project” action in the hero, services, projects, and footer.

### Rating

**Weighted overall score: 52/100 — Below professional standard.**

The homepage's visual execution is good; the score reflects the whole public experience, including empty project pages, weak conversion, incomplete internationalization, major SEO omissions, accessibility defects, and production security configuration.

## 2. Technical Audit

### Hosting, status codes, HTTPS, and redirects

- **Confirmed positive:** `https://atistat.com/` returned HTTP/2 `200`.
- **Confirmed positive:** `https://www.atistat.com/` returned a single `301` redirect to `https://atistat.com/`.
- **Critical:** `http://atistat.com/` returned `200 OK` and served the page over plaintext. Visitors following an old HTTP link are not upgraded and are exposed to interception or modification.
- **Confirmed positive:** A random nonexistent route returned HTTP/2 `404`.
- **Confirmed positive:** TLS verification succeeded using TLS 1.3. The certificate covered `atistat.com` and was valid from 12 June to 10 September 2026 at the time of testing.
- **Confirmed:** No mixed-content URL was found in the inspected markup; site assets are relative and the Google Maps iframe uses HTTPS.

**Required fix:** Configure an origin-level `301` or `308` redirect from every HTTP request to the corresponding HTTPS URL. Verify apex, `www`, path, query, and asset cases. Only after HTTPS is universal, deploy `Strict-Transport-Security`, initially with a conservative max-age and later with `includeSubDomains` if all subdomains qualify.

**Manual follow-up:** INP and field pass rates require real interaction data. One lab run cannot determine whether the site passes Core Web Vitals at the 75th percentile.

Specific causes and opportunities:

- **High:** The visible hero image is marked `loading="lazy"` in `index.html`. Lazy-loading the likely LCP candidate can postpone the most important image. Load it eagerly, add `fetchpriority="high"`, and preload only if measurement confirms benefit.
- **Medium:** None of the 18 homepage `img` elements has intrinsic `width` and `height`. The measured CLS was excellent, but dimensions improve deterministic layout and resilience.
- **Medium:** The site serves 1100-pixel project images of 203–314 KB into comparatively small timeline cards and supplies no `srcset` or `sizes`. Generate responsive derivatives.
- **Positive:** Image photography/artwork is WebP rather than JPEG/PNG, and below-fold images use native lazy loading.
- **Positive:** The custom font uses WOFF2 and `font-display: swap`.

### Compression, caching, and CDN

- **Measured high-impact issue:** The live homepage transferred approximately 98 KB, matching its uncompressed decoded size; no `Content-Encoding` was returned.
- **Measured:** The 26.6 KB CSS file also transferred uncompressed.
- **Measured:** CSS and image responses included `Last-Modified` and ETag validators but no explicit `Cache-Control` or `Expires`.
- **Observed:** Responses came directly from Apache; no CDN was evident from headers. A CDN is optional at this scale, but edge caching could improve geographic consistency.

**Fix:** Enable Brotli or gzip for HTML, CSS, JavaScript, SVG, JSON, XML, and fonts. Fingerprint versioned assets and serve them with long-lived immutable caching. Give HTML a shorter revalidation policy. Do not apply lossy recompression to existing WebP without visual review.

### HTML structure and validity

- **Positive:** Pages use a doctype, language attribute, viewport meta, header, navigation, main, sections, articles, headings, and footer.
- **Medium:** Header and footer embed the same exported SVG with the same IDs (`svg1`, `defs1`, `g1`, `layer2`, `layer3`, `namedview1`, `text1`, and `text2`). IDs must be unique in a document.
- **Low/medium:** The inline SVG begins with XML declarations inside HTML body content. These are unnecessary in inline HTML SVG and add parser noise.
- **Medium:** A project page is roughly 73 KB while its unique body content is only a heading and one word. The homepage is roughly 97.5 KB. Repeated inline WordPress styles and two very large inline logo SVGs dominate each document.
- **Low:** `<link rel="preconnect" href="">` is empty and provides no benefit.

**Fix:** Use one clean external SVG symbol or optimized logo asset, remove Inkscape metadata and XML declarations, give inline instances unique IDs if IDs remain, and dequeue unused WordPress block/global CSS on this custom theme.

### CSS and JavaScript quality

- **Positive:** The custom CSS has a coherent variable system, responsive breakpoints, focus styling for the timeline, and reduced-motion handling.
- **Positive:** The JavaScript is a strict, dependency-free IIFE and limits itself to navigation, reveal behavior, timeline control, and touch enhancement.
- **High resilience defect:** `.at-fade` is hidden by default in CSS. JavaScript later adds `.is-in`. If `main.js` is blocked, fails to load, or throws before reveal initialization, important content remains transparent. Visibility should be the default; hide only under a JavaScript-enabled class that is installed before paint, or use animation that never gates content.
- **Medium:** Mobile partner cards use a `data-href` click listener on `<li>` elements rather than real links. This is both an accessibility and maintainability problem.
- **Medium:** The timeline tab pattern mixes links and buttons inside `role="tablist"`, creating inconsistent semantics and behavior.
- **Observed:** No custom-code JavaScript error appeared in the Chrome run. Google Maps logged an informational console message from its own embed script.

### Broken links and resources

- **Confirmed positive:** A static crawl found the referenced local page assets and homepage fragments.
- **Confirmed positive:** `correctproject.com`, `engsys.bg`, and the Google Maps embed all resolved with final HTTP `200` during testing.
- **Caveat:** Query-named mirror artifacts such as `index.html?p=14.html` contain language-and-fragment combinations that do not map cleanly as literal static paths. These appear to be crawl-export artifacts rather than deliberate production URLs.
- **High discoverability issue:** The six project pages exist and are in the sitemap but are not linked from the visible project timeline. Project buttons reveal one-line panels; the two hero images link to `#opit`, not to their project pages.

### Robots, sitemap, URLs, canonicalization, and structured data

- **Positive:** The live `robots.txt` permits public crawling and points to an absolute sitemap URL.
- **Positive:** The live sitemap uses absolute URLs and lists the homepage, `/opit/`, and six project pages.
- **Medium:** `/opit/` is an English duplicate of the full homepage, yet appears as a separate indexable sitemap URL.
- **High:** The English homepage is not represented as a coherent language entry in the sitemap and no `hreflang` links were found.
- **High:** The root Bulgarian and English pages have no canonical link.
- **Medium:** Project canonicals exist, but mirror variants show inconsistent targets for some routes and query versions. Production canonical behavior should be normalized and tested with an actual crawler.
- **High:** No JSON-LD structured data was found.

### Forms, cookies, and error pages

- **Confirmed:** There are no public forms, login flows, checkout, account creation, or payment interfaces in the audited visitor pages. Form validation, CAPTCHA, password policy, payment security, and confirmation states are therefore not applicable to this front-end review.
- **Confirmed:** The public 404 returns the correct status and displays “Нищо не е намерено” (“Nothing found”).
- **Medium:** The 404 provides no explanatory copy, prominent home action, related navigation, contact route, or search alternative.
- **Confirmed:** No analytics or advertising script was found in the mirrored front end. Google Maps is the visible third-party embed.

## 3. SEO Audit

### On-page metadata

- **High:** Every inspected page lacks a meta description.
- **High:** No Open Graph title, description, image, type, or URL metadata was found.
- **High:** No Twitter Card metadata was found.
- **High:** No structured data was found for `Organization`, `LocalBusiness`, `Service`, `Person`, `BreadcrumbList`, or projects.
- **Medium:** Bulgarian and English homepages share the generic title “ATISTAT – Construction & Investment.” It does not describe the services, location, or language-specific intent.
- **Positive:** Project title elements are unique by project name.

Suggested homepage title:

Suggested Bulgarian description:

> АТИСТАТ развива и реализира жилищни, строителни и инвестиционни проекти в София — от анализ и планиране до строителство и въвеждане в експлоатация.

Copy must be checked against the exact operating geography and service scope before publication.

### Headings and search intent

- **Positive:** The homepage has one descriptive H1 followed by H2 section headings and H3 service/project headings.
- **Professional judgment:** The H1 is brand-led but only weakly search-intent-led. It says “Construction with professionalism, Investment with vision” without naming location, project development, renovation, or the desired customer.
- **High:** The project pages are thin content: each contains only a project name and a service phrase. They cannot satisfy project-research intent or demonstrate expertise.

### Internal linking and discoverability

- **Positive:** The main one-page sections are easy to reach through anchor navigation.
- **High:** Visible project cards do not link to the corresponding detail pages.
- **Medium:** The footer has no service, project, legal, or language navigation.
- **Medium:** The English site relies on a language switch but has no hreflang cluster.
- **Medium:** There are no breadcrumbs on project pages.

### International and local SEO

- **Positive:** Bulgarian and English homepages use appropriate `lang` attributes.
- **Confirmed failure:** English project files use `lang="en"` while their visible title and body remain Bulgarian. This is incorrect for search engines and assistive technology.
- **High:** No `hreflang="bg"`, `hreflang="en"`, or `x-default` implementation was found.
- **Positive:** Address, phone, business name, domain email, and map are visible.
- **Medium:** No organization/local-business schema, company registration identifier, business hours, or linked official profiles were found.

### E-E-A-T and authority

Credibility positives include the named founder, career chronology, project list, related-company links, award claim, address, phone, and business email. Missing authority evidence includes:

- Founder biography with qualifications and role.
- Team and named expertise.
- Company registration details and relevant construction registrations or certifications.
- Case-study facts, client/partner attribution where permitted, measurable scope, completion status, and documented outcomes.
- Direct evidence for the award claim.
- Testimonials, references, press, memberships, or safety/quality policies.

### SEO recommendation

Treat each project as a real landing page, not as an index stub. Give it a unique title and description, a canonical URL, correct language alternatives, breadcrumb schema, a concise project summary, location/year/status where publishable, ATISTAT's exact role, challenge, scope, delivery process, outcomes, optimized gallery, and a related-service CTA. Remove or redirect `/opit/` if it is not intentionally distinct. Submit a clean bilingual sitemap after deployment.

## 4. Content Audit

### What works

- The core proposition is understandable: construction plus investment expertise.
- The founder narrative gives the new brand a credible origin.
- The service steps are unusually specific for a short brochure site.
- The tone is restrained and professional rather than sales-heavy.
- Bulgarian and English homepage copy is broadly consistent.

### Weak or unclear content

1. **Generic claims without evidence.** Phrases such as “experience, responsibility and long-term vision,” “sustainable projects,” and “long-term value” recur without defining standards, methods, or results.
2. **No customer framing.** The site explains what ATISTAT does but not which client situations it solves: land evaluation, investor representation, full construction delivery, occupied-building renovation, or co-investment, for example.
3. **No differentiation.** Visitors cannot determine why ATISTAT is a better fit than another Bulgarian contractor or development partner.
4. **No project substance.** The six detail pages add almost nothing beyond the timeline.
5. **No next-step copy.** The site never sets expectations for an enquiry, required project information, response time, or first consultation.
6. **Incomplete English content.** Project “English” pages are Bulgarian.
7. **No risk-reduction content.** There is no explanation of budget control, schedule governance, quality assurance, procurement, reporting, change control, safety, warranty, or handover.
8. **No legal/privacy content.** The site includes contact information and a third-party map but no privacy notice or ownership/legal page.

### Representative improved copy

Current hero:

> Construction with professionalism, Investment with vision.

More specific direction:

> We build and develop enduring property projects — from site analysis and planning to construction and handover.

Supporting line:

> ATISTAT combines 26 years of construction and project-management experience for residential, reconstruction, and investment projects in Sofia.

Primary action:

> Discuss your project

Secondary action:

> Explore completed work

These examples show the needed specificity; they should not be published until the geographic scope and “26 years” claim are verified and a maintenance owner is assigned.

## 5. Visual and Aesthetic Audit

### Assessment

**Professional judgment:** The homepage is professionally designed, distinctive, and more premium than a typical small-company WordPress brochure. It does not look template-based. The project pages, however, look unfinished because their large empty canvas contains almost no content or imagery.

### Strengths

- Consistent cream, charcoal, and green palette aligned with architecture and investment.
- Custom Bookman-style typography gives the identity a mature editorial tone.
- Architectural line art creates a memorable, relevant image system.
- Good use of whitespace, dividers, numbering, and restrained motion.
- Desktop composition alternates image and copy effectively.
- Service layout and timeline create a clear visual hierarchy.
- Logo, images, and UI controls share a coherent visual language.

### Weaknesses

- **High:** The Bulgarian hero headline clips at the right edge in the 390-pixel narrow-window capture. The grid item retains a min-content width larger than the available column while `overflow-x: clip` hides the excess.
- The project pages are visually empty: no hero image, metadata, project facts, gallery, related work, or CTA.
- Some small green text fails normal-text contrast on cream.
- Mobile becomes an extremely long stack; all eight timeline cards precede the narrative and services, while no persistent or repeated conversion action helps the user.
- The same intricate logo SVG is inlined twice per page, creating technical bloat without visual benefit.
- The map's Google styling is visually less controlled than the rest of the identity.

### Recommended visual refinements

Preserve the current identity. It does not need a wholesale aesthetic replacement. Introduce a darker accessible green for small text, keep the current green for large display use where contrast passes, add a compact CTA component, create a factual project-detail system, and define mobile type/gutter rules that prevent min-content overflow. Use real project imagery selectively while retaining sketches as the brand's signature layer.

## 6. UX and Usability Audit

### Main journeys

#### First-time prospective client

The user lands on a strong hero, reads About, explores the timeline, reads four services, and reaches phone/email/map. The sequence is logical, but the user must scroll to the end to act. No button starts an enquiry, and no service explains who qualifies or what information to provide.

#### Credibility and portfolio review

The timeline names projects and responsibilities, but project controls only switch one-line panels. The existing project pages are not linked and contain no useful evidence. This journey ends before the visitor can validate experience.

#### English-language visitor

The homepage is usable in English. Moving to project content produces Bulgarian text marked as English. This breaks confidence in the bilingual promise.

#### Mobile visitor

The menu condenses correctly and content stacks, but the hero is clipped, the timeline becomes a long series of cards, click-only partner cards lose keyboard equivalence, and contact remains at the bottom of a roughly 9,200-pixel document.

### UX positives

- Simple four-item navigation.
- No intrusive pop-ups, autoplay, account wall, or banner.
- Telephone and email links invoke appropriate device actions.
- Map, address, phone, and email agree within the inspected pages.
- Current desktop timeline provides hover, focus, click, and arrow-key handling for actual tab buttons.
- Empty and loading states are largely irrelevant because content is server-rendered.

### Friction and dead ends

- No prominent CTA in the hero.
- Services have no actions or related case-study links.
- Project names do not open project details.
- Project detail pages have no back-to-projects control, related work, next/previous navigation, or CTA.
- New-tab external links are not announced visually.
- The 404 says only “Nothing found.”
- Search is absent; acceptable for the current small site, but a project filter would be more useful than global search once content expands.
- There is no enquiry form, so users must switch context to phone or email and compose an unstructured request.

## 7. Accessibility Audit

### Estimated level

**Partially compliant.**

The semantic foundation is better than many small sites, but confirmed failures in contrast, language, keyboard equivalence, hidden focusable controls, duplicate IDs, and mobile reflow prevent a “mostly compliant” rating.

### Confirmed strengths

- Page-level `lang` on the two home languages.
- Header, navigation, main, sections, articles, and footer.
- One homepage H1 and logical section headings.
- All 18 homepage images have non-empty alternative text.
- Navigation toggle has an accessible name, `aria-controls`, and `aria-expanded`.
- The map iframe has a title.
- Timeline buttons receive a visible custom focus outline.
- A reduced-motion media query prevents reveal and sketch motion.
- The mobile menu button is approximately a practical touch size.

### Confirmed failures and defects

1. **Text contrast:** `#538135` on `#f4f1e9` measures approximately 4.08:1 and on `#faf8f2` approximately 4.34:1. Both fail the WCAG AA 4.5:1 requirement for normal-sized text. The color is used for small language options, captions, labels, and other text.
2. **Incorrect skip-link name:** The skip link points to `#main` but is labelled “Меню”/“Menu,” not “Skip to main content.”
3. **Duplicate IDs:** Repeated header/footer SVG exports create multiple identical IDs in one page.
4. **Mobile partner cards:** `<li data-href>` cards are click handlers, not links or buttons. They are not keyboard focusable and have no semantic destination.
5. **Collapsed navigation focus:** At the mobile breakpoint the closed navigation has `max-height: 0` and `overflow: hidden`, but its links are not removed from the tab order with `hidden`, `inert`, or equivalent state management.
6. **Hidden panel focus:** Inactive timeline panels use `aria-hidden="true"` while some contain focusable links. Hiding a focusable descendant from the accessibility tree creates inconsistent keyboard/screen-reader behavior.
7. **Mixed tablist semantics:** A `role="tablist"` contains both actual tab buttons and ordinary external links. Panels are not consistently associated with labelled tabs.
8. **Language mismatch:** English project variants declare `lang="en"` while their visible content is Bulgarian.
9. **Mobile reflow:** The Bulgarian H1 visibly clips in the 390-pixel rendering. Hiding overflow does not satisfy reflow.
10. **JavaScript resilience:** Content hidden by `.at-fade` can remain unavailable if the script does not execute.

### Manual testing required

- Screen-reader reading order and announcement of timeline state.
- Full keyboard sequence at desktop and mobile widths.
- 200% and 400% zoom/reflow.
- Focus not obscured by the sticky header.
- Target sizes for language links and small inline controls.
- Google Maps keyboard behavior.
- Windows High Contrast/forced-colors behavior.
- Voice-control names for icon-only controls.
- Browser and assistive-technology combinations beyond Chrome.

## 10. Security and Privacy Review

### Confirmed positive findings

- Valid TLS certificate and TLS 1.3 on HTTPS.
- HTTP/2 available.
- No mixed content found in inspected pages.
- No public form, password, payment, or account data collection in scope.
- No advertising, analytics, or social-pixel script found in the mirror.
- External links opened in new tabs use `rel="noopener"`.

### Confirmed risks

- **Critical:** HTTP is not redirected to HTTPS.
- **High:** No HSTS header was present.
- **Medium:** No Content-Security-Policy, `X-Content-Type-Options`, `Referrer-Policy`, `Permissions-Policy`, or clickjacking protection was present in the observed home response.
- **Low:** `Server: Apache` exposes server family information.
- **Medium privacy risk:** Google Maps loads as a third-party iframe before any visitor choice, while the site supplies no privacy notice. Whether consent is legally required depends on actual storage, processing, jurisdiction, and configuration and needs qualified review.
- **Operational risk:** The public site is WordPress and exposes the standard REST index. This is normal behavior, not evidence of a vulnerability, but it reinforces the need for prompt core/theme/plugin updates, protected administration, least privilege, backups, logging, and rate controls.

### Recommended baseline

1. Force HTTPS.
2. Deploy HSTS after redirect validation.
3. Add `X-Content-Type-Options: nosniff`, a strict referrer policy, a suitable permissions policy, and clickjacking protection through CSP `frame-ancestors` or equivalent.
4. Develop CSP in report-only mode, including required Google Maps origins, before enforcement.
5. Publish a privacy notice and decide whether to use a click-to-load map.
6. Maintain WordPress updates, MFA for administrators, backups, malware/file-integrity monitoring, login throttling, and tested restoration.
7. Commission an authenticated WordPress security review if the business relies on the site; this audit did not test the admin area or origin for exploitable vulnerabilities.

### Information architecture

Recommended structure:

- Home
- Services
  - Residential construction
  - Renovation and reconstruction
  - Project development
  - Real-estate investment
- Projects
  - Filterable index by role/type/status only if the portfolio grows
  - Individual bilingual case studies

  - Founder and team
  - Approach, credentials, quality, and safety
- Contact
- Privacy / Legal

The first version can remain a fast WordPress theme without a page builder. Separate pages are warranted when they contain unique intent and evidence; otherwise keep concise sections on Home.

### Revised homepage structure

1. **Header:** Logo, Services, Projects, About, Contact, language switch, and a restrained “Discuss a project” button.
2. **Hero:** Specific outcome-led H1, one supporting sentence, primary enquiry CTA, secondary projects CTA, and the current signature sketch.
3. **Credibility strip:** Years of experience, project types, relevant registrations or awards—only verified facts.
4. **Selected projects:** Three high-quality linked case studies with role, year, project type, and one result.
5. **Services:** Four clear cards naming who each service is for, key scope, and a CTA.
6. **Approach:** A concise process from first review through handover, with quality, budget, schedule, and reporting principles.
7. **Founder/team:** Qualifications, roles, and related-company relationship.
8. **Proof:** Verifiable award, references/testimonials where permission exists, and memberships/registrations.
9. **Enquiry block:** What information to send, what happens next, response expectation, phone/email/form.
10. **Footer:** Business identity, EIK if appropriate, address, contact, navigation, language, privacy/legal, copyright.

### Design system

Retain the current palette, typography character, sketch motif, paper backgrounds, and restrained motion. Define:

- Accessible text colors separate from decorative brand green.
- A mobile type scale with safe long-word wrapping.
- An 8-pixel-based spacing scale.
- Buttons for primary, secondary, and text actions.
- Project cards with consistent facts and aspect ratios.
- Form controls with labels, help, error, success, and focus states.
- Breadcrumbs and simple pagination/related-project navigation.
- Image crops and responsive source rules.
- A no-motion and no-JavaScript baseline.

### Performance, accessibility, and conversion principles

- Never lazy-load the LCP image.
- Make HTML content complete before enhancement.
- Use real links and buttons, never click-only generic containers.
- Keep keyboard focus visible and remove hidden controls from focus order.
- Require correct page and passage language.
- Meet WCAG 2.2 AA contrast and reflow targets.
- Treat every project page as proof plus a next action.
- Measure field outcomes without adding intrusive tracking.

## 18. Final Verdict

ATISTAT does not currently meet a complete professional website standard, although its homepage visual design does. A complete rebuild is unnecessary. The strongest course is a **substantial partial redesign and production hardening** that preserves the identity and front-end restraint while replacing the thin content model, repairing accessibility/mobile defects, strengthening conversion, and completing security and SEO fundamentals.

The most important reason is that the site makes an excellent first visual impression but cannot substantiate it: its project pages contain virtually no evidence, and visitors have no intentional path from interest to enquiry.

Leaving the current problems unresolved risks lost high-value leads, weak search visibility, reduced confidence among investors and partners, exclusion of keyboard and some assistive-technology users, poor mobile perception, and avoidable transport-security exposure.

The recommended next step is a focused 30-day foundation release: enforce HTTPS, repair responsive/accessibility defects, add a conversion path, publish three evidence-rich bilingual case studies, and deploy a complete metadata/security baseline. Then expand the remaining portfolio and trust content through the same tested system.
